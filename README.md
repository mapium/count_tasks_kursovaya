# Система управления задачами (Task Management System)

RESTful API для управления задачами, сотрудниками и отделами организации. Система построена на FastAPI с использованием PostgreSQL в качестве базы данных.

## 📖 Описание

Система управления задачами предоставляет REST API для:
- Управления пользователями и аутентификации
- Управления сотрудниками организации
- Управления отделами с назначением руководителей
- Создания и управления задачами
- Отслеживания статусов задач
- Комментирования задач
- Группировки задач по отделам

Система поддерживает ролевую модель доступа с различными уровнями прав:
- **Администратор** (role_id = 1) - полный доступ
- **Руководитель отдела** (role_id = 2) - управление отделом
- **Сотрудник** (role_id = 3) - работа с собственными задачами

## 🛠 Технологии

- **Python 3.13+**
- **FastAPI** - современный веб-фреймворк для создания API
- **SQLModel** - ORM для работы с базой данных
- **PostgreSQL** - реляционная база данных
- **Argon2** - хеширование паролей
- **JWT** (python-jose) - токены для аутентификации
- **FastAPI Pagination** - пагинация результатов
- **Uvicorn** - ASGI сервер
- **Faker** - генерация тестовых данных
- **Alembic** - миграции базы данных
- **Pydantic** - валидация данных

## 🎲 Генерация тестовых данных

Для быстрого заполнения базы данных тестовыми данными используется скрипт `generate_data.py`, который создает реалистичные данные с помощью библиотеки Faker.

### Использование скрипта

1. **Убедитесь, что база данных создана и настроена**

2. **Запустите скрипт генерации данных**:
```bash
python generate_data.py
```

### Что генерируется

Скрипт автоматически создает следующие данные:

1. **Роли** (3 роли):
   - `admin` - Администратор системы
   - `manager` - Руководитель подразделения
   - `employee` - Сотрудник

2. **Статусы задач** (4 статуса):
   - К выполнению
   - В работе
   - На проверке
   - Выполнено

3. **Отделы** (5 отделов по умолчанию):
   - Отдел разработки
   - Отдел тестирования
   - Отдел продаж
   - Отдел маркетинга
   - HR отдел
   - Каждый отдел получает назначенного менеджера

4. **Пользователи** (20 пользователей по умолчанию):
   - 1 администратор:
     - Username: `admin`
     - Password: `admin123`
   - ~5 менеджеров (для управления отделами)
   - ~14 сотрудников
   - ⚠️ **Важно**: Все остальные пользователи имеют одинаковый пароль: `password123`

5. **Сотрудники** (20 сотрудников):
   - Полные ФИО (русские имена)
   - Телефоны в формате +7XXXXXXXXXX
   - Email адреса
   - Паспортные данные (10 цифр)
   - ИНН (12 цифр, опционально)
   - СНИЛС (11 цифр, опционально)
   - Привязка к отделам
   - Статус активности (75% активных)

6. **Задачи** (50 задач по умолчанию):
   - Название и описание
   - Создатель и исполнитель (из одного отдела)
   - Отдел исполнителя
   - Статус задачи
   - Приоритет (малый, средний, высокий)
   - Плановые даты начала и окончания
   - Фактические даты (если задача начата/завершена)

### Настройка количества данных

Вы можете изменить количество генерируемых данных, отредактировав функцию `main()` в файле `generate_data.py`:

```python
# Изменить количество отделов
departments_count = 5  # по умолчанию

# Изменить количество пользователей
users = generate_users(session, roles, count=20)

# Изменить количество сотрудников
employees = generate_employees(session, users, departments, count=20)

# Изменить количество задач
tasks = generate_tasks(session, users, employees, statuses, count=50)
```

### Особенности генерации

- **Русская локаль**: Faker использует локаль `ru_RU` для генерации реалистичных русских имен, адресов и текста
- **Уникальность**: Скрипт проверяет существование записей и не создает дубликаты для базовых данных (роли, статусы)
- **Связи**: Все связи между таблицами устанавливаются корректно с учетом внешних ключей
- **Логика данных**: 
  - Задачи создаются в отделах исполнителей
  - Менеджеры назначаются на отделы
  - Даты задач генерируются с учетом логики (плановая дата окончания после даты начала)

### Данные для входа после генерации

После выполнения скрипта вы получите:
- **Администратор**: `admin` / `admin123`
- **Остальные пользователи**: `<сгенерированный_username>` / `password123`

Примеры сгенерированных пользователей будут выведены в консоль после завершения работы скрипта.

## 📡 API Endpoints

### Аутентификация (`/api/v1/auth`)

| Метод | Endpoint | Описание | Требует авторизации |
|-------|----------|----------|---------------------|
| POST | `/auth/login` | Вход в систему, получение JWT токена | Нет |
| POST | `/auth/refresh` | Обновление токена доступа | Нет |

### Пользователи (`/api/v1/accounts` и `/api/v1/users`)

| Метод | Endpoint | Описание | Требует авторизации |
|-------|----------|----------|---------------------|
| POST | `/accounts/sign_up` | Регистрация нового пользователя | Нет |
| POST | `/accounts/sign_up_as_admin` | Регистрация пользователя администратором | Да (admin) |
| GET | `/users` | Получить список всех пользователей (с пагинацией) | Да (admin) |

### Сотрудники (`/api/v1/employees`)

| Метод | Endpoint | Описание | Требует авторизации |
|-------|----------|----------|---------------------|
| GET | `/employees` | Получить список всех сотрудников (с пагинацией) | Да (admin) |
| GET | `/employees/{id}` | Получить сотрудника по ID | Да (admin) |
| POST | `/employees` | Создать нового сотрудника | Да (admin) |
| PUT | `/employees/{id}` | Обновить данные сотрудника | Да (admin/manager/сам сотрудник) |
| DELETE | `/employees/{id}` | Удалить сотрудника | Да (admin) |

**Валидация при создании/обновлении сотрудника**:
- Email должен быть валидным адресом
- Паспортные данные: минимум 10 символов
- ИНН: минимум 12 символов (опционально)
- СНИЛС: минимум 11 символов (опционально)

**Права доступа**:
- Просмотр и создание: только администратор
- Обновление: администратор, менеджер отдела или сам сотрудник (только свои данные)
- Удаление: только администратор

### Отделы (`/api/v1/departments`)

| Метод | Endpoint | Описание | Требует авторизации |
|-------|----------|----------|---------------------|
| GET | `/departments` | Получить список всех отделов (с пагинацией) | Нет |
| GET | `/departments/{id}` | Получить отдел по ID | Нет |
| POST | `/departments` | Создать новый отдел | Да (admin) |
| PUT | `/departments/{id}` | Обновить данные отдела | Да (admin) |
| DELETE | `/departments/{id}` | Удалить отдел | Да (admin) |

**Особенности**:
- Просмотр отделов доступен без авторизации
- Создание, изменение и удаление: только администратор
- Отдел может иметь назначенного руководителя (`department_manager_id`)
- При удалении отдела проверяются связи с сотрудниками и задачами

### Задачи (`/api/v1/tasks`)

| Метод | Endpoint | Описание | Требует авторизации |
|-------|----------|----------|---------------------|
| GET | `/tasks` | Получить список всех задач, сгруппированных по подразделениям | Да (manager/admin) |
| GET | `/tasks/my-department` | Получить все задачи своего подразделения (с пагинацией) | Да |
| GET | `/tasks/my` | Получить все задачи текущего пользователя (с пагинацией) | Да |
| POST | `/tasks` | Создать новую задачу | Да (manager/admin) |
| PUT | `/tasks/{id}` | Обновить задачу | Да (manager/admin) |
| DELETE | `/tasks/{id}` | Удалить задачу | Да (manager/admin) |
| PATCH | `/tasks/{task_id}/status` | Изменить статус задачи | Да (создатель/исполнитель/admin/manager) |
| POST | `/tasks/{task_id}/comments` | Добавить комментарий к задаче | Да (создатель/исполнитель/admin/manager) |

**Особенности**:
- Задачи привязаны к отделам исполнителей
- Поддерживается группировка задач по отделам
- Приоритеты: "малый", "средний", "высокий"
- Статусы задач: "К выполнению", "В работе", "На проверке", "Выполнено"
- Изменение статуса доступно создателю, исполнителю, администратору или руководителю отдела
- Комментирование доступно создателю, исполнителю, администратору или руководителю отдела

**Права доступа**:
- Просмотр всех задач (сгруппированных): менеджер и выше
- Просмотр задач своего подразделения: любой авторизованный пользователь
- Просмотр своих задач: любой авторизованный пользователь
- Создание, изменение и удаление: менеджер и выше
- Изменение статуса и комментирование: создатель, исполнитель, администратор или руководитель отдела

## 🗄 Модели данных

### Users (Пользователи)
- `id` - уникальный идентификатор
- `username` - имя пользователя (уникальное)
- `password` - хеш пароля (Argon2)
- `role_id` - идентификатор роли

### Employees (Сотрудники)
- `id` - уникальный идентификатор
- `user_id` - связь с пользователем (опционально)
- `first_name`, `last_name`, `middle_name` - ФИО
- `phone_number` - номер телефона
- `email` - электронная почта (уникальная)
- `passport_data` - паспортные данные (уникальные, минимум 10 символов)
- `inn` - ИНН (уникальный, опционально, минимум 12 символов)
- `snils` - СНИЛС (уникальный, опционально, минимум 11 символов)
- `department_id` - отдел сотрудника
- `is_active` - статус активности

### Departments (Отделы)
- `id` - уникальный идентификатор
- `name` - название отдела
- `description` - описание отдела
- `created_at` - дата создания
- `department_manager_id` - идентификатор руководителя отдела (опционально)

### Tasks (Задачи)
- `id` - уникальный идентификатор
- `title` - заголовок задачи
- `description` - описание задачи
- `creator_id` - создатель задачи
- `assignee_id` - исполнитель задачи
- `department_id` - отдел задачи (отдел исполнителя)
- `status_id` - статус задачи
- `priority` - приоритет ("малый", "средний", "высокий")
- `planned_start_date` - запланированная дата начала
- `planned_end_date` - запланированная дата окончания
- `actual_start_date` - фактическая дата начала
- `actual_end_date` - фактическая дата окончания
- `created_at` - дата создания
- `updated_at` - дата обновления

### Task_Status (Статусы задач)
- `id` - уникальный идентификатор
- `name` - название статуса
- `order_index` - порядок сортировки

### TaskComments (Комментарии к задачам)
- `id` - уникальный идентификатор
- `task_id` - идентификатор задачи
- `author_id` - автор комментария
- `comment_text` - текст комментария
- `created_at` - дата создания

### Roles (Роли)
- `id` - уникальный идентификатор
- `name` - название роли (уникальное)
- `description` - описание роли

## 🔐 Аутентификация и авторизация

### Получение токена

Для получения JWT токена отправьте POST запрос на `/api/v1/auth/login`:

```bash
curl -X POST "http://localhost:8000/api/v1/auth/login" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "username=your_username&password=your_password"
```

Ответ:
```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "bearer"
}
```

### Использование токена

Добавьте токен в заголовок запроса:

```bash
curl -X GET "http://localhost:8000/api/v1/tasks" \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN"
```

### Роли и права доступа

- **Администратор (role_id = 1)**: Полный доступ ко всем функциям системы
- **Руководитель отдела (role_id = 2)**: Доступ к просмотру всех задач и управлению своим отделом
- **Сотрудник (role_id > 2)**: Доступ к работе с собственными задачами

## 💻 Разработка

### Структура кода

Проект следует принципам чистой архитектуры:
- **Models** - модели данных SQLModel
- **Schemas** - схемы валидации Pydantic с валидаторами
- **Controllers** - бизнес-логика
- **Routers** - маршрутизация API endpoints
- **Core** - общие утилиты (безопасность, конфигурация)
- **DB** - работа с базой данных

### Схемы валидации

Все входные данные валидируются с помощью Pydantic схем:
- `EmployeeCreate` / `EmployeeUpdate` - валидация данных сотрудников
- `TaskCreate` / `TaskStatusUpdate` - валидация данных задач
- `DepartmentsCreateSchema` - валидация данных отделов
- `UserSchema` / `UserSchemaCreate` - валидация данных пользователей

### Добавление новых endpoints

1. Создайте функцию в соответствующем контроллере
2. Добавьте роут в соответствующий роутер
3. Убедитесь, что роутер подключен в `main.py`
4. При необходимости создайте схемы валидации в `schemas/`

### Работа с базой данных

Все операции с БД выполняются через SQLModel сессии:
```python
from app.db.session import get_session
from sqlmodel import Session, select

def my_function(session: Session = Depends(get_session)):
    # Ваш код работы с БД
    pass
```

### Миграции базы данных

Проект использует Alembic для управления миграциями:

```bash
# Создание новой миграции
alembic revision --autogenerate -m "описание изменений"

# Применение миграций
alembic upgrade head

# Откат миграции
alembic downgrade -1
```

### Обработка ошибок

Все контроллеры используют единый подход к обработке ошибок:
- `HTTPException` для клиентских ошибок (400, 401, 403, 404)
- Откат транзакций при ошибках БД
- Логирование ошибок сервера (500)

## 📝 Примечания

- Пароли хранятся в виде хешей Argon2
- Все даты и время хранятся в UTC
- База данных автоматически создается при первом запуске через SQLModel
- Используется пагинация для больших списков (пользователи)
- Валидация данных выполняется на уровне схем Pydantic
- Приоритеты задач используют русские названия: "малый", "средний", "высокий"
